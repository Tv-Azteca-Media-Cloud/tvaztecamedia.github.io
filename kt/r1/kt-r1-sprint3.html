1.2Script Python — Idempotencia y Carga Masiva
Ejecución — Carga de Catálogo bash
# Ejecución normal python3 scripts/catalog-to-salesforce.py \ --org TVADevJosue \ --input catalog.xlsx # Dry-run — sin modificar la Org python3 scripts/catalog-to-salesforce.py \ --org TVADevJosue \ --input catalog.xlsx \ --dry-run # Cleanup — elimina specs no listados en CSVs aprobados python3 scripts/catalog-to-salesforce.py \ --org TVADevJosue \ --input catalog.xlsx \ --cleanup
Dependencias
Python 3.9+
Salesforce CLI 2.x
simple-salesforce
openpyxl
Flag --cleanup
Elimina AdSpaceCreativeSizeType, AdSpaceSpecProduct y AdSpaceSpecification cuyos IDs no estén en los CSVs aprobados.

Fases de ejecución del script:

  1. 1
    Preflight SOQL
    Carga el estado actual de la Org en memoria. Consulta objetos existentes: Product2, ProductCategory, AdSpaceSpecification y PricebookEntry.
  2. 2
    Calcula Hash GUID
    Genera un identificador único basado en campos técnicos del producto (nombre, categoría, plataforma, formato). Garantiza idempotencia entre ejecuciones.
  3. 3
    Inyecta TVA_GUID__c
    Graba el GUID calculado en el campo TVA_GUID__c en todos los objetos MC+RC del catálogo.
  4. 4
    Lógica Delta
    Si GUID existe: solo actualiza PricebookEntry si cambia la tarifa.
    Si GUID no existe: crea las 3 capas en bloque transaccional único.

Indicadores de salida:

+ created → Registro nuevo creado → skipped → Ya existía, no modificado ~ updated → Actualizado (ej. cambio de precio) ✗ ERROR → Falló la creación Summary X/Y → Total creados / total skipped

Columnas requeridas en el Excel:

ANo BPRODUCTO CCATEGORÍA DCOMERCIALIZACIÓN ETARIFA BASE FMEDIO GPLATAFORMA HUBICACIÓN IFORMATO JSERVIDORES

DRO Setup — Dynamic Revenue Orchestrator

3.1Qué es DRO
⚙
Dynamic Revenue Orchestrator
DRO es el módulo de Salesforce Media Cloud responsable de la gestión de aprovisionamiento y la conexión directa con Ad Servers externos.
Función 1
Descomposición de Orders
Descompone la Order en unidades de entrega individuales, mapeando cada Line Item a una unidad ejecutable en el Ad Server.
Función 2
Enrutamiento a Ad Server
Enruta cada unidad de entrega al Ad Server correspondiente (GAM, Advantage Serve) según la configuración del AdServerId.
Función 3
Confirmación de Aprovisionamiento
Confirma el aprovisionamiento vía REST API y recibe el Line Item ID del Ad Server para vincularlo en Salesforce.
Función 4
Monitoreo y Reconciliación
Monitorea la entrega contra el compromiso de la Order y reconcilia discrepancias durante el vuelo de la campaña.
3.2Settings DRO a Habilitar
Dynamic Revenue Orchestrator
Módulo principal de aprovisionamiento
ON
In-flight Amendments
Modificaciones durante vuelo de campaña — definición de flujos en Sprint 4
ON
3.3DRO Flows Configurados (Activos)
Flow Trigger Propósito
DRO Order Fulfillment Flow Al activar una Order Primer aprovisionamiento al Ad Server (GAM). Genera el Line Item y obtiene el ID de respuesta.
DRO Amendment Flow Modificación de una Order activa Gestiona cambios pre-vuelo o durante el vuelo de la campaña. Flujo definido en Sprint 4.

Integración Ad Server

Ad Server Integration — Google Ad Manager

Primera integración end-to-end operativa con Google Ad Manager. El DRO Order Fulfillment Flow envía el payload a GAM y recibe el Line Item ID que confirma el aprovisionamiento.

4.1GAM Integration Setup
Ad Server Type
Google Ad Manager (GAM)
Protocolo de conexión
REST API
Autenticación
OAuth 2.0 Service Account
Objeto de configuración
Registro Ad Server en Salesforce
🚫
Prerequisito Crítico — AdServerId en AdSpaceSpecification
AdServerId en AdSpaceSpecification es OBLIGATORIO. Sin él, el componente Ad Placement genera error en runtime. Debe estar configurado ANTES de Sprint 3 para que DRO funcione correctamente. Verificar en todos los registros de AdSpaceSpecification antes del aprovisionamiento.
4.2Aprovisionamiento Flow — Primer End-to-End
1
Order activada en Salesforce
El Sales Rep activa la Order en Salesforce. Status cambia de "Draft" a "Activated".
2
DRO Order Fulfillment Flow ejecutado
El trigger de activación dispara automáticamente el DRO Order Fulfillment Flow. Se construye el payload con los datos del Line Item.
3
Payload enviado a GAM REST API
Salesforce envía el payload de aprovisionamiento al endpoint REST de Google Ad Manager, autenticado con OAuth 2.0 Service Account.
4
GAM crea Line Item en el Ad Server
Google Ad Manager procesa el payload y crea el Line Item correspondiente en el sistema de ad serving.
5
GAM responde con Line Item ID
GAM devuelve el identificador único del Line Item creado. Este ID es el enlace entre Salesforce y el Ad Server.
6
Salesforce actualiza Fulfillment Status: "Provisioned"
El campo Fulfillment Status en el registro de Order se actualiza a "Provisioned", confirmando el aprovisionamiento exitoso.
7
Ad Order Item vinculado al GAM Line Item ID
El Ad Order Item en Salesforce queda vinculado al Line Item ID de GAM, habilitando el monitoreo y reconciliación posterior.
4.3Modelo de Datos Ad Server
Objeto Descripción
Ad Server Parámetros técnicos del servidor externo. Contiene URL, credenciales OAuth, tipo de Ad Server (GAM, Advantage Serve) y configuración de conexión.
Ad Server Account Vincula el Ad Server con una cuenta o marca específica del anunciante. Permite múltiples cuentas por Ad Server.
Ad Server Account Contact Conecta el Ad Server Account con el contacto de la agencia o anunciante responsable del Ad Server.
Ad Server User Mapea el Ad Server con el usuario interno de Salesforce (Media Planner o Sales Rep) que gestiona las campañas.

Arquitectura de Datos

Modelo de Datos — Bloque Productos y Precios

El catálogo de 12 productos se estructura sobre 7 objetos del modelo de datos de Salesforce Media Cloud. Cada capa tiene una responsabilidad específica en la jerarquía de producto.

Objeto Descripción y rol
Product Catalog Agrupación de alto nivel. Define el catálogo completo al que pertenecen los productos. Ejemplo: 'Catálogo Digital TV Azteca'.
ProductCategory Grupos temáticos dentro del catálogo: Streaming, Digital Display, Patrocinio, Servicios. Permite clasificación y filtrado en Browse Products.
Product2 Ítem vendible. Puede ser un producto simple o un paquete (Bundle). Es el objeto central del catálogo — nunca se elimina.
Product Classification Plantilla reutilizable de atributos compartidos entre productos de la misma categoría o tipo comercial.
Product Attribute Definition Variables configurables del producto: Rate Type (CPM/CPC/FIJO), Es omitible, Duración, Plataforma, Formato.
Price Book Standard Price Book — lista maestra de precios. Contiene todas las tarifas base del catálogo digital.
Price Book Entry Tarifa base de un producto específico en el Price Book. Actualizada por el script cuando el precio cambia (lógica delta).
