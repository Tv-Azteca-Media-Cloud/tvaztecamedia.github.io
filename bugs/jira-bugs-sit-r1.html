PR1066114-696 CPM — escala del Rate Type (bug de fórmula) Squad Media

CPM = Cost Per Mille — la unidad de Quantity en el modelo CPM es el millar por definición del modelo. Quantity = 1 en CPM debe equivaler a 1 000 impresiones, produciendo precio = CPM cotizado × 1. El sistema hoy trata 1 como 1 impresión aislada → precios 1000× menores. Es regresión contra la semántica OOTB del Pricing Element CPM (Revenue Cloud Advanced / Media Cloud on Core), no un pedido de "nuevo mínimo de 1000".

Próximo paso: inspeccionar Pricing Element / Price Rule / Pricing Procedure activo para productos con Rate Type CPM → identificar dónde se perdió la escala → corregir declarativamente en el step CPM (dividir Quantity por 1000, o usar Impressions Sold / 1000 × CPM) → validar dos casos-test → cerrar.
QA/SADivergencia doctrinal. QA/SA lo clasifica como "Mejora" con dudas ("???"): "No recuerdo requerimiento que contemple que las impresiones son 1000". FerR complementa: "también es por la fórmula del sector de publicidad (precio × cantidad / 1000)". Nuestra postura es que la escala /1000 es intrínseca al modelo Cost Per Mille — no es "mejora opcional", es la definición del Rate Type CPM. A confirmar con arquitectos Media: si el modelo OOTB de Media Cloud on Core aplica /1000 automáticamente en el Pricing Element CPM → bug real; si el proyecto configuró un Pricing Element genérico sin escala CPM → cae como mejora.
Reunión cliente 2026-07-22Divergencia resuelta a favor de bug real. Froy Rodríguez confirmó en la reunión que "el equipo de medios está realizando los ajustes necesarios en los tipos de tasas para corregir este comportamiento" — se reconoce el error en el campo de impresiones, donde se espera la unidad de medida de "1,000".
  • Posturadivergencia con QA/SA cerrada — ambos lados reconocen que la escala /1000 debe aplicarse. Sigue como bug real.
  • Soluciónajuste en los Rate Types del Pricing Element, en curso por el equipo de medios.
  • Validaciónno requiere validación adicional externa — Media Squad ya asumió la corrección.
