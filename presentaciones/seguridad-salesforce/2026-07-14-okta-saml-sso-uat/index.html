¿Qué es el ?sc=?

  • Es el ID interno del SSO Configuration en Salesforce (TVA_SSOOKTA)
  • Le indica a Salesforce cuál de los configs SAML activos debe usar para procesar el response
  • Sin él, Salesforce intenta resolver por Issuer — si falla → error 1601 y Login History vacío
  • Se obtiene en: Setup → Single Sign-On Settings → SSO config → sección Endpoints → Login URL
  • El ID es diferente en cada org — hay que obtenerlo del org de Producción
🔬
Por qué Login History estaba vacío
El error ocurría antes del login engine de Salesforce

Salesforce solo registra en Login History los intentos que pasan la validación básica del SAML Response. Si el SP no puede resolver el SSO config (por falta del ?sc=), descarta el request silenciosamente — antes del login engine — sin dejar rastro.

🚨
Síntoma observado
Login History vacío + SAML Tracer mostraba SamlError?idpError=1601
💡
Diagnóstico clave
Login History vacío = el error ocurre en el parsing del request, no en el login. Debugging desde el browser.

🚀 Plan de Acción — Producción
ℹ️
Lo que funcionó en UAT no se puede copiar directo a Producción.
El parámetro ?sc= es único por org — hay que obtenerlo del SSO config de Prod. Además hay dos requisitos de seguridad pendientes que el SP Metadata ya declara.
📋
Pasos en orden para activar SSO en Producción
Estimado: 1–2 horas · Requiere acceso admin a Salesforce Prod y Okta
  • 1
    Verificar que el SSO Config existe en Producción
    Setup → Single Sign-On Settings → confirmar que TVA_SSOOKTA (o equivalente) está creado en el org de Prod.
    Si no existe → crearlo con los mismos valores que UAT, usando el cert de Prod (ver paso 3).
    Si existe → continuar al paso 2.
  • 2
    Obtener el ?sc= del org de Producción
    Setup → Single Sign-On Settings → abrir el SSO config → sección Endpoints.
    Copiar la Login URL — el parámetro ?sc=XXXXXXXXXXXXXXX al final es el ID de Prod.
    Este valor es diferente al de UAT (0LEWr0000000Nfl) — no reutilizar.
  • 3
    Descargar el SP Metadata XML de Producción
    Setup → Single Sign-On Settings → SSO config → botón "Download Metadata".
    Este XML contiene el cert de Prod, la ACS URL con el ?sc= correcto, y los requisitos de firma y encriptación.
  • 4
    Importar el SP Metadata XML en Okta (app de Producción)
    Admin Console Okta → Applications → app Salesforce Prod → General → Edit SAML Settings.
    Buscar "Import metadata" o "Upload SP metadata" → subir el XML descargado.
    Okta configura automáticamente ACS URL (con ?sc=), Entity ID y ambos certificados.
    Si no hay opción de import: configurar manualmente con los valores del XML.
  • 5
    Activar Assertion Encryption en Okta Pendiente UAT también
    El SP Metadata declara KeyDescriptor use="encryption" → Okta debe encriptar la aserción.
    Show Advanced Settings → Assertion Encryption: Encrypted
    → Encryption Certificate: subir el cert TVA_OKTASSO.crt (o el equivalente de Prod)
    → Encryption Algorithm: AES256-CBC
  • 6
    Verificar Assertion Signing en Okta Pendiente UAT también
    El SP Metadata declara WantAssertionsSigned="true" → Okta debe firmar la aserción.
    En SAML Settings → Response: Signed y Assertion Signature: Signed.
    Verificar que el cert de firma de Okta (IdP cert) coincide con el que está en Salesforce.
  • 7
    Verificar Federation ID en los usuarios de Producción
    Setup → Users → abrir un usuario de prueba → campo Federation ID.
    Debe coincidir exactamente con el NameID que envía Okta (email o username de Okta).
    Si está vacío → el SSO falla aunque la configuración esté perfecta.
  • 8
    Validar con SAML Tracer antes del go-live
    Reproducir el flujo SSO con SAML Tracer abierto → confirmar:
    — El POST llega a la ACS URL con el ?sc= correcto de Prod
    — El SAMLResponse contiene <EncryptedAssertion> (si se activó encriptación)
    — Login History de SF registra el intento exitoso

📦 ¿Esta configuración va al repositorio?
💡
Respuesta corta: Parcialmente.
La configuración SSO de Okta+Salesforce es metadata de plataforma, no código. Parte va al repo como metadata desplegable; parte es configuración manual de Setup.

✅ Lo que SÍ va al repositorio

El certificado TVA_OKTASSO → como metadata tipo Certificate en force-app/main/default/certs/. Permite que el cert se despliegue automáticamente junto con el org.

El SSO Config TVA_SSOOKTA → como metadata tipo SamlSsoConfig en force-app/main/default/samlssoconfigs/. Incluye todos los parámetros excepto el contenido del cert del IdP (que referencia al cert por nombre).

Esto garantiza que si se crea un sandbox nuevo o se hace un refresh, la configuración se puede desplegar sin pasos manuales.

⚙️ Lo que es configuración manual de Setup

El cert del IdP (de Okta) → se sube en Setup → Single Sign-On Settings → Edit. Salesforce no permite desplegar certs de terceros vía metadata.

La activación del SSO en Session Settings → Setup → Session Settings → "Allow Identity providers to use certificates for authentication".

El Federation ID en los usuarios → es dato de cada usuario, no metadata. Se puede poblar vía Data Loader o script Apex de setup, pero no vía deploy de metadata.

📁
Estructura sugerida en el repo
Siguiendo SFDX source format
force-app/main/default/ ├── certs/ │ └── TVA_OKTASSO.crt-meta.xml ← metadata del certificado (sin clave privada) └── samlssoconfigs/ └── TVA_SSOOKTA.samlssoconfig ← configuración SSO completa
⚠️
La clave privada del cert nunca va al repo. Solo el certificado público (.crt-meta.xml referencia el cert pero no incluye la llave privada). La llave privada vive únicamente en Salesforce Certificate and Key Management.

💡 Consideraciones de Seguridad y Acceso
🔒
Acceso de admins durante la activación de Okta
Cómo evitar quedar bloqueados durante la configuración
Nivel 1 — Single Sign-On Settings (org-wide)
El checkbox Disable login with Salesforce credentials debe permanecer desmarcado durante toda la fase UAT.
Nivel 2 — Permiso a nivel de Perfil (requerido)
Setup → Profiles → [Perfil Admin] → System Permissions → Edit
Desactivar: Is Single Sign-On Enabled
→ Con esto desactivado, los admins siguen entrando con usuario/contraseña aunque el SSO esté activo.
📋
Acción antes de activar OKTA en UAT: Verificar que el perfil de los admins tenga Is Single Sign-On Enabled desmarcado. Garantiza acceso de respaldo por credenciales nativas.
🛡️
Alerta MFA de Salesforce — No hay conflicto con Okta
Okta MFA adaptativa satisface el requerimiento MFA de Salesforce
Qué pide Salesforce

Que todos los usuarios autentiquen con un segundo factor (MFA) al acceder. Aplica a todas las orgs.

Qué provee OKTA

OKTA ya incluye MFA adaptativa en el flujo SAML 2.0. El MFA de OKTA satisface completamente el requerimiento de Salesforce.

  • Salesforce acepta como MFA válido cualquier segundo factor gestionado por un IdP externo de confianza (como OKTA) dentro del flujo SSO.
  • El flujo SAML 2.0 incluye el paso de autenticación OKTA (Password + MFA Push/TOTP) antes de redirigir al usuario a Salesforce.
  • No se requiere activar el MFA nativo de Salesforce cuando OKTA provee el factor adicional en el IdP.
📋
Excepción a considerar: Los admins que entren con credenciales nativas durante UAT sí necesitarían MFA nativo de Salesforce activado en su perfil.
