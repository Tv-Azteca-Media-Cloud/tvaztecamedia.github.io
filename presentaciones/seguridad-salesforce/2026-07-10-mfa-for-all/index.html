¿Qué está cambiando?
Resumen del cambio aplicado por Salesforce

Salesforce está aplicando técnicamente el Multi-Factor Authentication (MFA) para todos los logins de empleados (usuarios internos), tanto en acceso directo a la UI como a través de Single Sign-On (SSO), en orgs de producción y sandbox.

Lo que Salesforce activa automáticamente

  • El setting "Require MFA for all direct UI logins" se activa y queda bloqueado — no se puede desactivar.
  • El permiso "Waive MFA for Exempt Users" deja de funcionar automáticamente; ahora requiere aprobación de Salesforce Support.
  • Orgs trial que se conviertan a suscripción ya no tienen 30 días de gracia.

Experiencia del usuario al hacer login

  • Si el usuario no tiene MFA registrado, se le forzará a registrar un método antes de entrar.
  • El flujo de login cambia a username primero → luego contraseña o passkey (two-step).
  • Se recomienda activar Passwordless Login con Passkeys para la experiencia más rápida y segura.
👥
¿A quiénes afecta?
Usuarios incluidos y excluidos del requisito
⚠️
SÍ aplica (usuarios afectados)
  • Todos los usuarios internos con Standard User License que hacen login directo a la UI o por SSO.
  • Usuarios que acceden a Employee Community o sitios de Experience Cloud con licencia de usuario interno.
  • Usuarios Chatter Plus (se consideran internos).
  • Usuarios con login directo que no tienen ninguno de estos permisos: System Administrator Profile, Modify All Data, View All Data, Customize Application, Author Apex — estos ya tienen requisito más estricto de Phishing-Resistant MFA.
✅
NO aplica (exentos)
  • Usuarios externos con Experience Cloud User Licenses (clientes, portales).
  • Usuarios Chatter External y Chatter Free.
  • Usuarios de SSO cuyo IdP ya envía señal válida de AMR o ACR de MFA estándar o phishing-resistant.
  • Usuarios que ya tienen MFA registrado y tienen el permiso "MFA for UI Logins" activo.
  • Developer Edition, Trial, Scratch orgs y orgs no pagadas.
  • Logins por API (solo afecta UI logins).
  • Connected Apps con flujos JWT Bearer o Client Credentials.

🔵 Impacto específico si usas OKTA como Identity Provider (IdP)

Si TV Azteca utiliza OKTA para SSO, el requisito de MFA puede cumplirse sin cambios en Salesforce — pero solo si OKTA está enviando correctamente las señales AMR o ACR en el token. De lo contrario, los usuarios verán una pantalla de registro de MFA en Salesforce al hacer login.

✅
Si OKTA ya hace MFA — esto es suficiente siempre que…
  • OKTA esté configurado para exigir MFA a los usuarios antes de pasar la aserción a Salesforce.
  • El token SAML o OIDC incluya la señal okta_verify en el campo AMR — o — un valor como mfa, rsa, timesynctoken en ACR.
  • Salesforce detecta esa señal y considera al usuario como Standard MFA — no le pide volver a autenticarse.
⚠️
Si OKTA hace MFA pero NO envía la señal — hay problema
  • Salesforce trata cualquier login SSO sin señal AMR/ACR como "No MFA".
  • El usuario verá una pantalla en Salesforce para registrar un método MFA adicional.
  • Esto ocurre aunque OKTA ya haya pedido MFA — Salesforce no lo sabe si no recibe la señal.
Valores AMR reconocidos para OKTA (Standard MFA)
SeñalSignificado¿Dónde configurar en OKTA?
okta_verifyOkta Verify App (push o TOTP)Enviado automáticamente por Okta en AMR cuando se usa Okta Verify
mfaMFA genérico estándarOKTA lo incluye en AMR por defecto si MFA fue requerido
timesynctokenTOTP (tiempo sincronizado)Cuando el usuario usa Google Auth o similar vía OKTA
fido / fido2Phishing-resistant (WebAuthn)Si el usuario usa FIDO2/WebAuthn en OKTA
phrPhishing-resistant MFASi se usa hardware key o passkey en OKTA
Cómo verificar que OKTA envía la señal correcta
  • Para OIDC: en Salesforce ir a Setup → Login History y revisar la columna Authentication Method Reference — debe mostrar okta_verify, mfa u otro valor válido.
  • Para SAML: usar un SAML tracer (extensión de browser) e inspeccionar el response — buscar el atributo con nombre que contenga amr o authnmethodsreferences.
  • En OKTA Admin Console: Security → Identity Providers → tu app de Salesforce — verificar que el MFA policy esté activo y que el claim AMR esté habilitado en el token.
⚠️
Nota sobre SAML ACR: Salesforce aún no registra valores ACR de SAML en Login History. Para verificar SAML ACR debes usar un SAML tracer externo. El validador de SAML de Salesforce solo clasifica como "strong" o "weak" — no refleja los nuevos tiers de este cambio.
🔗
Señales SSO: cómo evalúa Salesforce AMR y ACR
Lógica de evaluación por tipo de protocolo
ProtocoloTipo de señalCómo se evalúaEjemplo de valor válido
SAML AMR Salesforce busca atributos cuyo nombre contenga amr o authnmethodsreferences. Soporta valores separados por punto y coma (hwk;mfa). Valores con URN/URL se dividen por : o /. okta_verify; mfa; hwk;face;mfa
OIDC AMR Es un array de strings con comparación exacta. ["okta_verify", "mfa"]
SAML / OIDC ACR Usa contains() sobre el string en minúsculas — tokens cortos como mfa hacen match en URNs largos. urn:oasis:names:tc:SAML:2.0:ac:classes:mfa
ℹ️
Cualquier valor de la categoría Phishing-Resistant satisface automáticamente también el nivel Standard MFA. Es suficiente que el IdP envíe al menos un valor reconocido en la columna AMR o ACR correspondiente.

Acciones recomendadas para TV Azteca
Qué hacer antes y después del enforcement
Antes del enforcement
Inmediato
Auditar usuarios con "Waive MFA" permission: Identificar todos los usuarios y perfiles que tienen este permiso activo. Ver SOQL en la sección inferior. Contactar a Salesforce Support si se necesita conservar la exención para herramientas de testing automatizado.
Inmediato
Verificar configuración de OKTA: Confirmar que el SAML response o el OIDC token incluye señales AMR/ACR válidas. Revisar Login History en Sandbox para verificar que las señales lleguen correctamente.
Esta semana
Habilitar métodos MFA en Setup: Ir a Setup → Identity Verification y activar Built-in Authenticators (Passkeys) para dar a los usuarios la opción de registrar el método más seguro. Nota: una vez habilitado, está disponible para todos los usuarios — no se puede restringir solo a admins.
Esta semana
Probar en Sandbox primero: Activar MFA en sandbox y validar el flujo de registro de usuarios antes de que llegue el enforcement a producción.
Antes del 20 Jul
Comunicar a usuarios: Avisar a todos los usuarios internos sobre el cambio y promoverles que registren su método MFA antes de la fecha límite para evitar interrupciones al hacer login.
Antes del 20 Jul
Establecer procedimientos de recuperación de acceso: Definir el proceso interno para que los admins puedan generar códigos de verificación temporales para usuarios bloqueados, sin depender de Salesforce Support.
Revisar
Revisar Connected Apps y automatizaciones: Los flujos OAuth Web Server y Hybrid Token requieren UI login y están sujetos a MFA. Los flujos JWT Bearer y Client Credentials no se ven afectados. Actualizar los tests automatizados que simulan el flujo de login.
Después del enforcement — resolución de problemas
  • Usuarios bloqueados: El Admin puede generar un código de verificación temporal desde Setup → Users → [usuario] → Generate Identity Verification Code.
  • SSO sin señal AMR/ACR: Coordinar con el equipo de OKTA para que el claim correspondiente se incluya en el token. Mientras tanto, los usuarios deberán registrar MFA adicional en Salesforce.
  • Monitoreo continuo: Revisar Login History para identificar usuarios que no pueden completar el challenge de MFA.

SOQL de auditoría
Consultas para identificar usuarios con el permiso "Waive MFA"

El campo API correspondiente al permiso "Waive Multi-Factor Authentication for Exempt Users" es PermissionsBypassMFAForUILogins.

Opción 1 — Usuarios con el permiso asignado vía Permission Set

SELECT AssigneeId, Assignee.Name, Assignee.Email, Assignee.Username, Assignee.IsActive, PermissionSet.Name FROM PermissionSetAssignment WHERE PermissionSet.PermissionsBypassMFAForUILogins = true

Opción 2 — Perfiles con el permiso habilitado

SELECT Id, Name, PermissionsBypassMFAForUILogins FROM Profile WHERE PermissionsBypassMFAForUILogins = true
⚠️
Cualquier usuario o perfil que aparezca en estas consultas dejará de estar exento a partir del enforcement. Si hay casos válidos (testing automatizado), contactar a Salesforce Support antes de la fecha límite para solicitar la exención formal.
