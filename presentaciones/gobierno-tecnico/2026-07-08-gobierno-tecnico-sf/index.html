Política de Refresh de Sandboxes

🔄 Dev Sandboxes (DEVURI, DEVMBU…)
  • Refresh on demand por el developer responsable
  • Datos de prueba generados con scripts de DataFactory
  • No requieren aprobación — responsabilidad del dev
  • Sin datos de producción en ningún caso
🔄 Build / QA (BLDCOM, QA)
  • Refresh al inicio de cada sprint
  • Pipeline CI/CD automático vía GitHub Actions
  • Aprobación del TA requerida para refresh manual
  • Scripts de validación automáticos post-deploy
🔄 SIT / STGPRE
  • Refresh por cada release (antes de SIT)
  • Aprobación conjunta TA + Release Manager
  • Datos de prueba representativos del volumen productivo
  • STGPRE sincronizado con PROD post go-live
🔄 UAT (Full Copy)
  • Refresh Full Copy antes de cada release
  • Aprobación: TA + Release Manager + PO de TV Azteca
  • Datos reales de producción — acceso controlado
  • Ventana de refresh: mínimo 48h antes del inicio de UAT
⚠️
Modo On-Preview (Salesforce releases): Cuando Salesforce tiene un release próximo (3 veces al año: Spring, Summer, Winter), los sandboxes creados o refrescados pueden activarse en modo On-Preview para recibir la nueva versión antes de producción. Sin ese modo, el sandbox quedaría en la versión anterior y requeriría un Sandbox Refresh posterior — perdiendo toda configuración realizada. Para UAT y SIT, siempre coordinar el refresh con el calendario de releases de Salesforce.

Checklist del Technical Architect — Revisión de PR

✅ El TA aprueba la PR solo si...
✓ El nombre de rama sigue el patrón feature/[JIRA-ID]
✓ Todos los campos y clases usan el prefijo correcto del squad (TVAM_, TVAS_, TVAI_, TVA_)
✓ Nomenclatura PascalCase en metadata — sin underscores, sin espacios
✓ Descripción obligatoria en todo artefacto nuevo: [SQUAD] | [JIRA-ID] | Propósito
✓ Cobertura de pruebas >90% — sin DML en capa de servicios
✓ Sin SOQL ni DML dentro de loops
✓ Todo SOQL tiene LIMIT — sin queries sin límite sobre objetos grandes
✓ Sin IDs hardcodeados — usar RecordTypeUtil o Custom Metadata
✓ Toda clase Apex con with sharing o without sharing explícito
✓ Triggers siguen el Trigger Action Framework — 1 trigger por objeto, lógica en Action Classes
✓ El cambio solo toca la carpeta del squad correspondiente — sin cross-squad sin autorización
✓ CI/CD pasó sin errores — validación en BLDCOM exitosa
🚫 Bloqueantes — PR rechazada automáticamente
✗ Cobertura de tests por debajo del 75% (mínimo Salesforce) — meta interna: 90%
✗ CI falla en validación — errores de compilación o test failures
✗ SOQL o DML dentro de un for loop
✗ IDs de Record Type u OrgId hardcodeados en el código
✗ Prefijo incorrecto o metadatos en la carpeta del squad equivocado
✗ Nombre de clase o trigger sin sufijo estándar (Test, Trigger, Batch, etc.)
⏱ SLA de Revisión
PRs estándar Revisión en 24 horas hábiles por el TA
PRs urgentes Mismo día — coordinar con TA directamente
Hotfix producción Proceso fast-track — rama hotfix/vX.X.X, revisión inmediata

Principio Central — Configuration First Approach

Configuration First

Antes de escribir una sola línea de código, pregunta si Salesforce puede resolverlo con configuración. Los desarrolladores somos el último recurso, no el primero.

✓Máxima compatibilidad con futuros releases de Salesforce
✓Menor deuda técnica y tiempo de mantenimiento
✓Admins pueden mantener sin desarrollo custom
→Código custom = último recurso, bien justificado

2 Declaration · Flows / Process Builder
3 AppExchange Package · Low-Code

  • No se permite hardcoding contra Profiles o Permission Sets

  • Los requerimientos de información se definen temprano con consulta a la función de gestión de información

  • Reducir complejidad capturando datos una vez y compartiéndolos con otras áreas
  • Los errores en datos deben corregirse en la fuente y distribuirse

  • Cada decisión de diseño identificada debe documentarse como Design Record

  • El uso de Salesforce Classic para usuarios finales no está permitido ni se mantiene

Trigger Action Framework — Arquitectura de Triggers

Principios del Framework
1
Adiós a los Handlers Gigantes
Principio de Responsabilidad Única mediante clases de acción pequeñas y atómicas (una acción = una clase).
2
Orquestación por Metadata
El orden de ejecución se configura directamente en Custom Metadata Trigger_Action_mdt, no se programa en código.
3
Coexistencia Híbrida (Apex/Flow)
El código dispara Autolaunched Flows para mantener orden. Before-Save Flows exclusivamente para actualizaciones rápidas de campos.
4

Saltar validaciones pesadas en cargas masivas usando Custom Permissions desde la configuración.
5
Testing de "Acción"
Pruebas unitarias enfocadas en la lógica aislada construyendo datos en memoria (Cero DML).

Apex Best Practices

⚡ Bulkification
  • No SOQL ni DML dentro de un loop — nunca
  • Usar SOQL for loop para conjuntos grandes de registros
  • Bulkify helper methods — manejar 200 registros
  • Usar Collections y For Loops eficientes
  • Un solo Trigger por objeto — siempre bulkificado
🛡️ Error Handling
  • Nunca catch vacío ni solo System.debug(e.getMessage())
  • Bloques catch deben: crear mensaje de error, lanzar excepción más específica, o usar sObject.addError()
  • Multi-phase DML: usar save points con try/catch y rollback
  • Iterar saveResults para verificar errores en Database methods
🔍 SOQL / DML
  • Sin IDs hardcodeados — usar RecordTypeUtil
  • Statement DML para sintaxis simple (all-or-none)
  • Method DML si algunos registros pueden fallar
  • Siempre LIMIT en queries sobre objetos grandes
  • ORDER BY campo indexado + LIMIT en LDV
  • No SOQL sin LIMIT — regla absoluta
🏗️ Controller
  • Usar controller extensions sobre custom controllers a menos que sea estrictamente necesario
  • Usar transient para datos que no persisten en view state
  • Controlar cantidad de registros retornados — implementar paging
  • Errores esperados como page messages con call to action
  • Errores inesperados en catch como page message (no mostrar excepción del sistema)

Large Data Volumes (LDV) — Best Practices

⚠️
Contexto TV Azteca: Con múltiples temporadas de pautas (5+ años de historial en Vista 360), miles de Line Items de GAM y sincronización de contratos de MDS, los objetos Order, OrderItem, TVAM_AdQuoteItem__c y Contract pueden crecer rápidamente a LDV. Estas prácticas son obligatorias desde el primer sprint.
📐 Diseño de Objetos para LDV
→Agregar ORDER BY campo indexado + LIMIT a todo SOQL sobre objetos grandes
→Usar hard delete cuando sea posible — el recycle bin ocupa espacio de almacenamiento
→Evitar NULL en campos indexados — reemplazar con literal (picklists o FK)
→Minimizar campos en el SELECT — solo los necesarios (narrow queries)
→Mantener el número de registros retornados al mínimo (selective queries) — usar filtros por campos indexados
→Queries en batches para procesamiento masivo — Database.QueryLocator en Batch Apex
🏷️ Campos a Indexar en TVA
Los campos usados frecuentemente en WHERE y ORDER BY deben estar indexados. Salesforce indexa automáticamente Id, Name, OwnerId, RecordTypeId y campos con External ID o Unique.
TVAI_IdExterno__c External ID ✓
TVAI_GAMDealId__c Marcar External ID
TVAI_GAMOrderId__c Marcar External ID
CreatedDate / LastModifiedDate Auto-indexado ✓
TVAI_StatusIntegracion__c Picklist — no indexable
❌ Anti-patrón — evitar en LDV
// Sin LIMIT — puede retornar millones de rows SELECT Id, Name, Amount FROM Order WHERE Status = 'Activated'; // NULL en campo indexado — query no selectiva SELECT Id FROM TVAM_AdQuoteItem__c WHERE TVAI_GAMOrderId__c = null; // Todos los campos — carga innecesaria SELECT * FROM Contract;
✅ Patrón correcto para LDV
// ORDER BY indexado + LIMIT siempre SELECT Id, Name, TVAI_IdExterno__c FROM Order WHERE Status = 'Activated' ORDER BY CreatedDate DESC LIMIT 200; // Usar literal en lugar de NULL SELECT Id FROM TVAM_AdQuoteItem__c WHERE TVAI_GAMOrderId__c = '' AND TVAI_GAMDealId__c != '' LIMIT 20;

Estándar de Error Handling y Retry Logic

Niveles de manejo de error
HTTP 2xx — Éxito
Loguear el response. Actualizar campo de estado en el registro. Continuar el flujo.
HTTP 401 / 403 — Auth Error
Refrescar token automáticamente y reintentar 1 vez. Si falla de nuevo → error permanente.
HTTP 5xx / Timeout — Error transitorio
Retry con backoff exponencial: 1 min → 5 min → 15 min (máx 3 intentos). Después → notificar AdOps.
HTTP 4xx (no 401) — Error de negocio
No reintentar. Loguear payload + response. Notificar al equipo de integración inmediatamente.
Campos de auditoría obligatorios por integración
→TVAI_Log[Sistema]__c — Long Text Area con último JSON
→TVAI_StatusIntegracion__c — Picklist: Pending / Sent / Error / Retry
→TVAI_FechaUltimoIntento__c — DateTime del último callout
→TVAI_NumeroIntentos__c — Integer para controlar retries
Notificación de errores — RNF-001
1.Error persiste después de 3 retries → crear Task para AdOps con contexto del fallo
2.Errores críticos (5xx sostenido) → Email Alert al Squad Integration
3.Nunca mostrar error técnico crudo al usuario final — mensaje amigable siempre

Governor Limits · Apex Jobs · Debug Logs · Setup Audit Trail · Event Monitoring · Escalación
Riesgo sin monitoreo
Un error silencioso en la integración MDS puede dejar órdenes sin enviarse. Un job fallido de GAM Polling deja deals programáticos sin confirmar. Un Apex con CPU limit alcanzado interrumpe el flujo de cotización sin aviso al usuario.
Qué monitorear
⚙ Governor Limits en tiempo real
🔄 Jobs asíncronos (Batch / Queueable / Sched)
🔌 Callouts a MDS, GAM y Data Cloud
🔐 Accesos y cambios de configuración
🐛 Errores Apex en producción
Principio
Los errores deben detectarse antes que el usuario final. El equipo técnico debe enterarse de un fallo de integración por una alerta automática, nunca por un reporte del cliente.
