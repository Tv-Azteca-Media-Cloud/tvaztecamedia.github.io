MEDIA-001 Accepted Media Layer Data Cloud
Data Cloud — Nested Payload Deserialization
in Bronze→Silver Batch Data Transforms
Fecha: 2026-07-10 Layer: Media Scope: Salesforce Data Cloud Relación: First ADR of media layer
🗺Context

Media/ad-tech connectors ingested into Salesforce Data Cloud frequently deliver structured payloads inside a single string field: XML fragments, nested JSON objects, or aggregated arrays. Two concrete cases exist in the project today:

  • GAM Connector — the packaged Google Ad Manager connector materializes LineItems_TVA_GAM_Connection__dll with the field Stats__c containing an XML fragment (<stats><impressionsDelivered>…</impressionsDelivered>…</stats>). Other columns of the same DLO (DeliveryIndicator__c, CustomTargetingAggregate__c, GrpSettings__c, FrequencyCaps__c, DeliveryData__c) follow the same pattern.
  • Anticipated future connectors — Marketing Cloud webhooks, S3 landing files, and Ingestion API integrations for other media-domain systems will produce similar nested payloads (JSON envelopes containing arrays of counts, XML SOAP responses, etc.).

Without a canonical location for the parsing operation, each downstream consumer (DMO field mappings, Calculated Insights, LWCs querying via GraphQL, reports) will re-implement the deserialization independently. This leads to:

  • Regex/parse logic duplicated across the platform
  • Inconsistent NULL semantics
  • Silent schema drift when the source payload changes shape
  • Re-parsing cost paid on every read
Scope
This decision applies to Data Cloud only. Payload transformations happening upstream in an integration orchestrator (per ADR-001 Passive Core) or in Apex are governed by other decisions.
✅Decision
Canonical Rule

Deserialization of nested string payloads inside Data Cloud occurs in a Batch Data Transform (BDT) that reads from a Bronze DLO and writes to a Silver DLO. The Silver DLO contains only the join key plus the parsed fields — never a full copy of the Bronze row.

📋Layer Contract
Layer Role Payload State
Bronze Raw ingest (connector output, Ingestion API landing) Nested payload preserved as-is (XML/JSON in string)
Silver Parsed, typed, per-source Flat columns, no nested payloads
Gold Canonical DMO (harmonized across sources) Only flat inputs from Silver
Reporting Calculated Insights / Segments / Activation Consumes Gold
🔷Silver DLO Shape

The Silver DLO produced by the BDT must contain:

  1. The join key(s) to the Bronze DLO (typically the natural business identifier — e.g. LineItemId__c for GAM Line Items).
  2. Optionally, a source watermark (e.g. SourceLastModified__c) used as Event Time for upsert semantics.
  3. The parsed fields derived from the payload, typed appropriately (BIGINT, DECIMAL, TIMESTAMP, etc.).
Important constraint
The Silver DLO does not duplicate the remaining Bronze columns. Downstream consumers that need Bronze columns join to Bronze on the shared key.
🚫Anti-patterns Explicitly Rejected
DLO Formula Fields on Bronze
Formula fields recalculate on every read — no materialization, no incremental processing, no schema validation. Acceptable for ad-hoc exploration in Data Explorer; not acceptable as a production contract.
Parsing inside DMO Field Mapping
Couples parse logic to the canonical model, prevents reuse when a second consumer needs the same fields without the DMO, and mixes source-format concerns with harmonization concerns.
Parsing inside Calculated Insights
CI is a Gold-layer aggregation surface. Doing deserialization in a CI mixes layer responsibilities and produces CIs that are brittle to source-format change.
BDTs that Deserialize AND Aggregate
A BDT that parses Stats__c and simultaneously aggregates to a weekly rollup mixes Silver (parse) with Gold (aggregate). Split them: one BDT Bronze→Silver, one CI or BDT Silver→Gold.
🏷Naming Convention

Silver DLOs produced by a deserialization BDT are named:

<SourceEntity><Concept>_<SourceSystem>__dll

For example, for the parsed Stats__c of GAM Line Items:

LineItemStats_TVA_GAM__dll
Why <Concept>?
The <Concept> component makes explicit which sub-structure of the Bronze row is being materialized. A single Bronze DLO may have multiple Silver DLOs, one per parsed payload column.
🔀Alternatives Considered
Deserialize at ingestion time (Ingestion API + Apex/middleware)

Rejected as the general rule because the project's primary media data source (GAM) is ingested via the packaged GAM Connector, which cannot be customized to run an Apex parser inline. This alternative remains valid for future sources ingested via Ingestion API — but it is out of scope for this ADR because it lives upstream of Data Cloud, governed by the integration orchestrator (ADR-001).

DLO Formula Fields with REGEXP_EXTRACT

Rejected — non-materialized, read-time cost, no schema validation, poor reuse. Acceptable for one-off exploration in Data Explorer only.

DMO Mapping Expression Parsing

Rejected — couples deserialization to the canonical model. If a second consumer (e.g. a per-source LWC) needs the parsed fields but not the harmonized DMO, the parse logic must be duplicated. Also produces a DMO whose mapping semantics are load-bearing on source-format details, which violates the "canonical model is source-agnostic" property.

Skip Silver — Ingest Bronze directly into Gold DMOs

Rejected — forces the DMO mapping to parse the payload (the previously rejected DMO-mapping anti-pattern) and prevents per-source materialization.

⚖️Consequences
✅ Positive
  • Single source of truth for parse logic per payload. All downstream consumers read from the Silver DLO — parse logic lives in one BDT, versioned as DataTransform metadata.
  • Read cost paid once. BDT materializes the result; consumers pay only their query cost against a flat, typed DLO.
  • Explicit contract between ingestion and modeling layers.
  • Schema drift produces a localized failure at the BDT — not scattered across N mapping expressions.
  • Testable in isolation. The BDT can be validated against sample Bronze rows independently of any DMO or CI.
⚠️ Negative
  • Storage duplication scoped to key + parsed fields only. Bounded by parsed columns per row (~6 fields for GAM Stats, negligible on ~157k row DLO).
  • Recurring BDT credit consumption (~1–2 credits/day estimated for the current GAM LineItems volume). Acceptable trade-off.
  • Schema-drift monitoring obligation. New source fields don't flow to Silver automatically — team must monitor connector release notes.
  • Silver DLO becomes a versioned dependency. Schema evolution must be treated as a public API change.
↩Reversibility

Partially reversible; cost grows with adoption. Reverting the pattern requires:

  1. Reimplementing the parse in the alternative location (DMO mapping / CI / Formula).
  2. Redirecting all consumers away from the Silver DLO.
  3. Retiring the BDT and Silver DLO.
Adoption note
Cost is bounded and mechanical when there is one consumer; grows meaningfully as consumers multiply. Early commitment is correct because reversal cost grows with time.
🔍Verifiability

Compliance can be verified during Phase 4 (Review) via:

1
No parse functions inside DMO field mapping expressions
Grep DMO metadata — any match on parse functions inside a mapping expression is a violation.
# Flag violations in DMO mapping expressions grep -r "REGEXP_EXTRACT\|REGEXP_SUBSTR\|PARSE_JSON" \ --include="*.dataModelObject-meta.xml" .
2
No Formula Fields on Bronze DLOs performing structural parses
Grep DLO Formula Field definitions for parse functions — flag for review; acceptable only if the field is documented as exploratory-only.
3
Every Bronze DLO column holding a nested payload has a Silver DLO downstream
For each Bronze DLO with a string column containing XML/JSON (identified by inspection or Data Explorer sampling), a corresponding BDT must exist. Absence of the BDT is a governance gap.
4
Silver DLO naming convention
Files must include the Concept naming (<Source><Concept>_<System>__dll) — never a hybrid that hides the layer role.
# Check naming convention on Silver DLOs find . -path "*/dataLakeObjects/**/*_TVA_*__dll.md"
5
No BDT performs deserialization AND aggregation in the same transform
Review BDT nodes: a BDT with both a REGEXP_EXTRACT-style node and a GROUP BY aggregation is a violation.
🔗Related ADRs
ADR-001
Passive Core
Governs whether parsing should happen upstream of Data Cloud (in an integration orchestrator) vs. inside Data Cloud. This ADR applies once the payload is inside Data Cloud; deserialization done by the orchestrator before ingestion is out of scope here.
ADR-004
Standard Data Model First
The Silver DLO is a per-source intermediate, not a substitute for a canonical DMO. The Gold DMO layer remains the standard-first canonical model; Silver serves only as the deserialization boundary.
Relationship to base layer
First ADR of the media industry layer. Does not specialize any base ADR — introduces a new Data Cloud governance dimension not covered by existing base decisions. If the base layer later adopts a general Data Cloud transformation strategy ADR, this one may become a specialization of it.
